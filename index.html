Git Diff Analogy
COW operations are like a git diff for partitions. Instead of storing the full new system (900 MB), you store just the patch (what changed). The full new system is reconstructed on-the-fly by applying the patch on top of the old system.
Key: Virtual A/B never permanently duplicates partition data. slot_b is a temporary, virtual construct that exists only until the merge completes.
payload.bin — Classic A/B vs Virtual A/B
payload.bin IS different for VAB. It contains new metadata that Classic A/B payload never had.
Who Generates payload.bin?
# Runs on Google/OEM build servers — NOT on device delta_generator \ --new_image=system_android14.img \ --old_image=system_android13.img \ --out_file=payload.bin \ --cow_version=2 # tells generator to include VAB metadata
Classic A/B payload.bin Contents
# payload.bin structure (Classic A/B) manifest: partition: system operations: - type: SOURCE_COPY # copy blocks from old system_a to new system_b - type: BSDIFF # binary diff patch (old → new) - type: PUFFDIFF # puffin patch for zlib compressed content - type: REPLACE_BZ # brand new compressed blocks - type: ZERO # zero out blocks # NO estimate_cow_size field # NO cow_version field # update_engine writes directly to physical system_b
Virtual A/B payload.bin Contents
# payload.bin structure (Virtual A/B) manifest: partition: system cow_version: 2 # NEW! VAB flag estimate_cow_size: 204800000 # NEW! pre-calculated bytes operations: - type: SOURCE_COPY # → translated to COW_COPY by update_engine - type: REPLACE # → translated to COW_REPLACE - type: ZERO # → translated to COW_ZERO - type: SOURCE_BSDIFF # → decomposed into COW_COPY + COW_REPLACE # estimate_cow_size is CRITICAL: # SnapshotManager uses it to pre-allocate COW file on /data # Without this → SnapshotManager can't reserve space!
estimate_cow_size — How It's Calculated
💡 Analogy: payload.bin for VAB is like a renovation blueprint that includes a materials estimate — it describes what to change AND tells the contractor (SnapshotManager) exactly how much storage to reserve before work begins.
# delta_generator computes estimate_cow_size like this: estimate_cow_size = (num_REPLACE_ops × block_size_4KB) # raw block data storage + (num_COPY_ops × sizeof(CowCopyOp)) # tiny struct ~16 bytes each + (num_ZERO_ops × sizeof(CowZeroOp)) # tiny struct ~8 bytes each + COW_header_size # fixed overhead + 10% safety_buffer # alignment + rounding margin # Example for Android 13→14 update of 900MB system partition: # 50,000 REPLACE ops = 50,000 × 4KB = 200 MB # 10,000 COPY ops = 10,000 × 16B = 160 KB # 5,000 ZERO ops = 5,000 × 8B = 40 KB # Safety buffer (10%) = 20 MB # estimate_cow_size ≈ 220 MB # (vs 900MB for full copy in Classic A/B!)
Key difference: update_engine does NOT write directly to a physical partition in VAB. It writes to a virtual dm-snapshot device, which internally routes writes to the COW file. update_engine doesn't know or care about the COW file — SnapshotManager abstracts that away.
COW Operations — The Building Blocks
Copy-on-Write (COW) operations are the "git diff" language of Virtual A/B. Only changes are stored — not the full partition.
💡 COW Strategy: "Don't copy the entire data upfront. Only record WHAT CHANGED. Reconstruct the full picture on demand by combining old partition + COW operations."
COW_REPLACE Full Block Replacement
Block has completely new data that cannot be derived from existing blocks. The full 4KB of new content is stored in the COW file. Used when block content changed significantly (e.g., updated binary, changed framework code).
StoresFull 4KB new block content
Storage cost4KB per operation (most expensive)
Use caseFramework JAR updated, native library changed
Old block[1]: [framework.jar v13.0 bytes...] New block[1]: [framework.jar v14.0 bytes...] COW stores → COW_REPLACE { dest_block: 1, data: [new 4KB] } At read time: block[1] → return new 4KB from COW file ✓
COW_COPY Block Relocation / Move
New block content is identical to an existing block at a different location in system_a. Instead of storing the data again, just store (source_block, dest_block) — a tiny 16-byte struct. Used when files/data are rearranged/moved in the new partition layout.
StoresJust source_block + dest_block (~16 bytes)
Storage cost~16 bytes (extremely cheap!)
Use caseFile moved to different location in partition
block[5] in new system = block[2] in old system_a (file relocated) COW stores → COW_COPY { src_block: 2, dest_block: 5 } At read time: block[5] → read block[2] from system_a origin ✓
COW_ZERO Block Zeroing
Block should become all zeros in the new system. No data needs to be stored — just record which block gets zeroed. Used when a file is deleted or a region is cleared in the new partition.
StoresJust dest_block (~8 bytes)
Storage cost~8 bytes (almost free!)
Use caseFile deleted, region wiped in new partition
block[7] should be zeros in new system COW stores → COW_ZERO { dest_block: 7 } At read time: block[7] → return all zeros ✓
COW_LABEL Checkpoint / Resume Marker
Not a data operation — a sequence number marker written periodically into the COW file. If the device loses power during OTA, update_engine reads the last COW_LABEL to find where to resume. Critical for OTA resumability (especially important in AAOS where car might be turned off mid-update).
StoresSequence number (checkpoint ID)
Storage cost~8 bytes
Use casePower loss recovery — resume OTA from last checkpoint
update_engine writes every N blocks: COW stores → COW_LABEL { sequence: 4271 } On resume after power loss: read COW_LABEL → sequence: 4271 → resume from operation 4271, skip 0..4270 ✓
Concrete Example — system_a → system_b delta
# system_a has 225,000 blocks (900MB partition, 4KB each) # Old system_a blocks: block[0]: [Android 13 init code ] ← unchanged block[1]: [framework.jar v13.0 ] ← CHANGED block[2]: [libutils.so v13.0 ] ← MOVED to block[5] block[3]: [build.prop version=13 ] ← CHANGED block[4]: [unchanged driver code ] ← unchanged block[5]: [old file data ] ← ZEROED (file deleted) ... block[175000]: [unchanged media codec] ← unchanged # Resulting COW file for system_a: COW_REPLACE { block:1, data:[new framework.jar bytes] } → 4KB COW_REPLACE { block:3, data:[version=14\n...] } → 4KB COW_COPY { src:2, dst:5 } → 16B COW_ZERO { block:2 } → 8B # blocks 0, 4, 175000, etc. have NO COW entry → read from system_a # COW file total: ~8KB + headers # vs full system_b copy: 900MB # Savings: >99.9% in this example!

Why COW File Needs a Loop Device Adapter
Problem: dm-snapshot is a kernel Device Mapper target — it can only connect to block devices. But the COW file lives on the /data filesystem — it's a regular file. The kernel cannot use a filesystem file as a block device directly.
# The adapter chain: /data/gsi/ota/system_cow (regular filesystem file, pre-allocated 200MB) ↓ [IImageManager / loop device] ← SnapshotManager sets this up # Uses dm-bow (Block Device Wrapper) or loop device # to present the file AS a block device ↓ /dev/block/mapper/system_a-cow-img ← NOW it's a block device! ↓ [dm-snapshot can use it] ✓ # Analogy: dm-snapshot is a plumber who only connects PIPES (block devices). # The COW file is water in a TANK (filesystem file). # The loop device is the ADAPTER converting tank → pipe fitting.
How dm-snapshot Serves Read Requests
# dm-snapshot block read logic (pseudocode): function read_block(block_number): if COW_file.has_entry(block_number): if entry.type == COW_REPLACE: return entry.data # new data from COW elif entry.type == COW_COPY: return system_a.read(entry.src_block) # redirect to source elif entry.type == COW_ZERO: return zeros(4096) # return zeros else: return system_a.read(block_number) # unchanged → read origin # This lookup is O(log n) using an in-memory hash table of COW entries # After first access, Linux page cache serves repeated reads from RAM

Space Accounting — What Takes Real Flash Space?
Component RAM Flash (/super) Flash (/data) Notes
/dev/block/mapper/system_b-snapshot ~few KB 0 ✓ 0 ✓ Pure virtual — just a DM mapping table
COW file /data/gsi/ota/system_cow 0 0 ✓ ~200MB ⚠ Only real space consumed. Temporary — deleted after merge
dm-snapshot kernel structures ~few KB 0 ✓ 0 ✓ Kernel data structures only

Snapshot Metadata — /metadata Partition
Why /metadata exists, what it stores, the boot sequence timing, and the state machine that drives the entire VAB lifecycle.
The Amnesia Problem — Why /metadata Must Exist
🧠 Analogy: Imagine you fall asleep (device reboots). When you wake up (boot), you have complete amnesia. You remember NOTHING about what happened before sleep.

Questions you need answered the moment you wake up — before you even get out of bed:
❓ "Was I in the middle of updating my house?"
❓ "Which rooms were being renovated?"
❓ "Where are the renovation blueprints stored?"
❓ "Is it safe to enter the house normally?"

You need a NOTE left on your bedside table that you can read THE MOMENT you wake up — before accessing any other room.

/metadata IS that bedside table note. 📝
⚠️ The Core Conflict: COW files live on /data. But /data is FBE encrypted — it cannot be read until the user unlocks OR CE keys are available. But /system needs to be mounted BEFORE that. So init needs "reconstruct snapshot" instructions BEFORE it can even access /data. That's exactly what /metadata solves.
Boot Sequence — What's Available WHEN
T=0ms Kernel starts, ueventd creates /dev/block/* symlinks T=50ms /metadata mounted ← FIRST partition mounted! Always. No encryption. first_stage_init reads: /metadata/ota/snapshot_boot = "1" ← Am I in snapshot mode? ← /system NOT MOUNTED YET ← /data NOT MOUNTED YET (FBE encrypted!) ← /vendor NOT MOUNTED YET T=100ms first_stage_init acts on /metadata: IF snapshot_boot = 1: reads /metadata/ota/snapshots/* for each partition calls SnapshotManager → rebuilds dm-snapshot devices sets up dm-verity on top of dm-snapshot mounts /dev/block/dm-X → /system ← virtual system_b! ELSE: mounts /system normally (no snapshot logic) T=200ms second_stage_init starts (from /system/bin/init) starts services, Zygote, etc. T=500ms /data mounted and decrypted ← COW files NOW accessible (BUT TOO LATE — /system was already mounted at T=100ms!) # This is WHY /metadata must hold snapshot config — # /data becomes available 400ms+ after /system is already mounted!
Key insight: If snapshot_boot=0 in /metadata → init skips ALL snapshot logic entirely. Zero performance cost on normal boots. The flag is only set to 1 during an active OTA, and cleared back to 0 after merge completes.
Concrete — What /metadata Tells first_stage_init Step by Step
SCENARIO: Device reboots after OTA applied. first_stage_init starts. Has NO memory of anything. Reading /metadata for first time. Step 1: init mounts /metadata (always first, always unencrypted) Step 2: reads /metadata/ota/snapshot_boot = "1" → YES, I am in snapshot mode! Don't boot normally. If value were "0" → skip all snapshot logic, boot normally. Step 3: reads /metadata/ota/snapshots/system_a: cow_file_size: 204800000 ← "COW file is 200MB on /data" snapshot_size: 943718400 ← "virtual device should be 900MB" status: UNVERIFIED ← "not yet verified boot" reads /metadata/ota/snapshots/vendor_a: cow_file_size: 52428800 ← "vendor COW is 50MB" Step 4: SnapshotManager reconstructs dm-snapshot: origin = /dev/block/mapper/system_a (real data) cow = /data/gsi/ota/system_cow (200MB COW) size = 900MB → creates /dev/block/mapper/system_b-snapshot ✓ Step 5: dm-verity set up on system_b-snapshot → /dev/block/dm-X Step 6: mount /dev/block/dm-X → /system ✓ Device now running from virtual slot_b (system_a + COW)
/metadata as the OTA Logbook — Updated Throughout the Process
/metadata is NOT written once — update_engine and SnapshotManager keep updating it as the OTA progresses, so after any reboot (power loss, user reboot, etc.) the device knows EXACTLY where in the process it was.
update_engine starts OTA: /metadata/ota/update_state = "INITIATED" /metadata/ota/snapshot_boot = "0" (not ready yet) OTA writing to COW file complete: /metadata/ota/update_state = "UNVERIFIED" /metadata/ota/snapshot_boot = "1" ← NOW set! Next boot will use snapshot. First successful boot into virtual slot_b: /metadata/ota/update_state = "VERIFIED" Merge starts (background): /metadata/ota/update_state = "MERGING" Merge complete, COW files deleted: /metadata/ota/update_state = "MERGE_COMPLETED" /metadata/ota/snapshot_boot = "0" ← cleared! Next boot is normal. → Next boot: init reads snapshot_boot=0 → skips snapshot entirely ✓
💡 One-line summary: /metadata is the ONLY partition that is small, unencrypted, and mounted before everything else — making it the ONLY safe place to store "what OTA state are we in?" so first_stage_init can reconstruct virtual partitions before mounting /system, even though COW files are locked behind /data encryption.

  • Mounts /dev/block/dm-X as /system

VOLD's Role in Merge Safety — dm-bow Checkpoint
⚠️ The Merge Risk: During merge, COW ops are written into system_a. If power is lost midway — system_a is partially written (half old, half new data). system_a is now CORRUPTED. Device is BRICKED!
# VOLD's checkpoint solution (dm-bow — Block Device Wrapper): BEFORE MERGE: VOLD::startCheckpoint() # Enables dm-bow on /data filesystem # Any write to /data is now journaled/checkpointed # If power loss → /data rolls back to last checkpoint # system_a is protected — merge hasn't started yet DURING MERGE (background): # SnapshotManager writes COW blocks into system_a # If power loss here → on next boot: # dm-bow detects incomplete journal → rolls back /data # system_a is still clean → falls back to snapshot reads # Merge resumes from where it left off AFTER SUCCESSFUL MERGE: VOLD::commitChanges() # Marks checkpoint as committed # Disables dm-bow overhead # COW files deleted # /metadata update_state = MERGE_COMPLETED ON FAILURE: VOLD::abortChanges() # Rolls back any partial /data writes # Keeps snapshot alive → device still bootable
Common Misconceptions & Corrections
Key corrections from our discussion — these are exactly the kind of subtle points that come up in Staff Engineer interviews.
Misconception 1 — "update_engine creates the COW file"
❌ WRONG: update_engine creates COW device in /data, then asks SnapshotManager to create virtual partition. ✅ CORRECT: SnapshotManager creates the COW file in /data. update_engine calls SnapshotManager::CreateUpdateSnapshots() and SnapshotManager does ALL of: - Creating COW file via fallocate() - Exposing it as block device via IImageManager - Creating dm-snapshot kernel target - Returning the virtual device path to update_engine update_engine never directly touches /data or creates files there. SnapshotManager owns all COW file management.
Misconception 2 — "update_engine applies COW ops FROM COW device TO snapshot"
❌ WRONG: update_engine reads COW operations from COW device and applies them to snapshot device. ✅ CORRECT: update_engine WRITES new block data TO the snapshot device. The snapshot device (dm-snapshot) INTERNALLY routes those writes into the COW file. Flow: update_engine → write block 42 → /dev/block/mapper/system_b-snapshot dm-snapshot kernel target intercepts write dm-snapshot → stores block 42's new data in COW file automatically update_engine never knows about the COW file!
Misconception 3 — "No slot switch in VAB, always merges to system_a"
❌ WRONG: In VAB, there is no slot switch. The update is always merged back into system_a. ✅ CORRECT: BOTH slot switch AND merge happen — at different times! Timeline: 1. OTA applied to virtual slot_b snapshot 2. Bootloader metadata → SWITCHED to slot_b ← slot switch DOES happen 3. Device reboots into virtual slot_b (snapshot = system_a + COW) 4. After successful boot → MERGE starts in background 5. COW ops written back into system_a → system_a becomes new version 6. COW files deleted 7. Next reboot: boots slot_b but slot_b now = updated system_a (no COW) The slot switch is real. The merge just happens AFTER the switch, converting system_a to contain the new data so slot_b doesn't need the COW overlay anymore.
Misconception 4 — "vbmeta stores the full hash of each partition"
❌ WRONG: vbmeta stores SHA256(entire /system partition) as a single hash. ✅ CORRECT: For LARGE partitions (system, vendor, odm): vbmeta stores the ROOT HASH of the Merkle tree (top-most hash of the hash tree embedded in the partition) NOT a direct hash of all 2GB For SMALL partitions (boot, dtbo): vbmeta stores SHA256(entire partition) — direct full hash Small partitions are cheap enough to hash fully. Why this matters: If vbmeta stored full partition hash → must hash ALL 2GB at boot → slow! With Merkle root hash → boot just trusts the root hash (already signed) dm-verity verifies individual blocks lazily at read time using the tree.
Misconception 5 — "dm-verity hashes the entire partition at setup time"
❌ WRONG: When dm-verity target is created, it hashes the entire dm-linear device. ✅ CORRECT: dm-verity does NOT hash anything at setup time. It is LAZY — it only verifies a block WHEN that block is actually read. The hash tree is PRE-BUILT at build time (by the build system) and PRE-EMBEDDED at the end of the partition image. At setup time: init just tells kernel "use THIS root hash from vbmeta" At read time: dm-verity intercepts the block read, computes hash, walks the tree, compares to root hash — all on-demand.
When Does dm-snapshot read_block() Execute?
A common confusion — read_block() runs at NEITHER OTA write time NOR merge time. It runs at a third time entirely.
🎯 Direct Answer: read_block() executes during Phase 2 — runtime, when the device is booted into virtual slot_b and ANY process reads ANY file from /system or /vendor. It is the live serving mechanism, not part of writing or merging.
The Three Phases — What Happens in Each
PHASE 1 — OTA WRITING (update_engine applies payload.bin): ══════════════════════════════════════════════════════════ update_engine WRITES blocks → dm-snapshot → COW file read_block() is NOT called here. Only WRITE path is active. dm-snapshot intercepts writes: "new block 42 data" → stored into COW file on /data system_a is never read or touched during this phase. COW file is being POPULATED with new data. PHASE 2 — RUNTIME (device booted into virtual slot_b): ══════════════════════════════════════════════════════════ THIS IS WHEN read_block() RUNS. Continuously. For every file access. App opens /system/framework/framework.jar: ext4 filesystem → "give me block 1" dm-snapshot intercepts → read_block(1): block 1 in COW? YES (COW_REPLACE) → return new jar data ✓ App opens /system/lib/libc.so (unchanged file): ext4 → "give me block 100" dm-snapshot → read_block(100): block 100 in COW? The Customer Service Analogy
read_block() = Customer Service Counter

Runs every day the store (device) is open.

Merge = Renovation crew permanently moving stock from the back room (COW) to the main shelf (system_a). They don't go through the customer service counter — they have direct access.

GPT Updates, VAB Partition Coverage & boot Size Growth
Can GPT be updated via OTA? Which partitions does VAB cover? What happens if boot grows too large?
Can We Update the GPT Table via OTA?
⚠️ Physical GPT update via OTA: Almost NEVER done — extremely risky. But logical partition changes inside /super: YES, freely via OTA.
# GPT (GUID Partition Table) lives at flash LBA 0 and LBA 1 # Contains: list of ALL physical partitions with name, start LBA, end LBA, GUID Flash layout: ┌────────┬────────┬────────┬───────────────────────┬──────────┬──────────┐ │ GPT │ boot_a │ boot_b │ /super │ userdata │ metadata │ │ LBA 0 │ │ │ [system][vendor][odm] │ │ │ └────────┴────────┴────────┴───────────────────────┴──────────┴──────────┘ ↑ Fixed at factory flash time. Changing this = moving walls while people live in the building.
Why Physical GPT Updates Are Dangerous
RISK 1Bricking — No Recovery
GPT lives at LBA 0. If power fails WHILE writing GPT → partition table is corrupted → bootloader can't find ANY partition → device is a permanent brick. No A/B fallback can save you (GPT is pre-A/B — it's what defines A/B!).
RISK 2No Rollback Possible
GPT has no slot_a/slot_b concept. There is only ONE GPT. If update fails → no way to go back. Everything else in Android has A/B rollback — but the GPT itself is the foundation that makes A/B possible.
RISK 3Zero Free Space Between Partitions
Flash is fully partitioned at factory time. Adding a new physical partition means needing unallocated space between existing partitions — which typically doesn't exist. Can't add new partition without shrinking another (risks data loss).
RISK 4Bootloader Dependency
Qualcomm ABL/XBL reads GPT directly. If you change GPT without updating bootloader simultaneously → bootloader looks for "boot" at old LBA → finds wrong data → device won't boot. Requires atomic GPT+bootloader update which is extremely complex.
How OEMs Add "New Partitions" via OTA — Dynamic Partitions
💡 Analogy: Physical GPT is like the city zoning map — set in stone, very hard to change. But /super is like a large empty plot inside the city. You can build new buildings (logical partitions) inside that plot freely, without rezoning the whole city.
# /super physical partition: fixed in GPT, NEVER changes # BUT logical partitions INSIDE /super: fully flexible via OTA! LpMetadata (stored at start of /super — THIS is updatable via OTA): partitions: system_a: offset=0, size=900MB vendor_a: offset=900MB, size=300MB odm_a: offset=1200MB, size=100MB system_ext_a: offset=1300MB, size=200MB ← NEW partition added via OTA! # OTA update can: # ADD new logical partition (e.g., system_ext) ✓ # REMOVE a logical partition ✓ # RESIZE a logical partition (grow/shrink) ✓ # All WITHOUT touching the physical GPT! ✓ Real example — Android 11 added system_ext: Android 10 /super: [system_a][vendor_a][odm_a][FREE SPACE] OTA to Android 11: [system_a][vendor_a][odm_a][system_ext_a][FREE] Physical GPT: ← UNCHANGED
When IS Physical GPT Updated? (Rare Cases)
Factory re-flashfastboot flash — entire device wiped, GPT rewritten from scratch. OK because device is in bootloader mode.
retrofit_dynamic_partitionsSpecial one-time OTA migrating legacy → dynamic partitions. Requires fastbootd + userspace fastboot. Done once during platform migration.
Qualcomm AAOS (rare)Some platforms reserve "gpt_backup" partition. 2-stage update: Stage 1 = update bootloader + backup GPT. Stage 2 = update GPT with recovery fallback. Extremely platform-specific.

What If boot Partition Size Increases After Update?
Physical boot_a and boot_b partition sizes are FIXED in GPT at factory time. They cannot grow via OTA. If a new Android version needs a bigger boot image than the physical partition allows → OTA fails safely. The active slot is completely untouched.
Does This Scenario Actually Exist? YES — History of boot Growth:
# Real boot image growth over Android versions: Android 9: boot.img = kernel + ramdisk ~32MB Android 10: boot.img = kernel + ramdisk (recovery merged) ~48MB ← grew! Android 12: boot.img = kernel + generic ramdisk ~32MB (split!) vendor_boot.img = vendor ramdisk + DTBs ~32MB ← new! Android 13: boot.img = kernel ONLY ~20MB init_boot.img = generic ramdisk ~8MB ← new split! # Each split was PARTLY motivated by physical partition size constraints! # "Can't make boot bigger → split contents across multiple smaller partitions"
What Happens If New boot Image > Physical Partition Size?
# Scenario: factory allocated boot_a/boot_b = 64MB each # New OTA has boot.img = 70MB (6MB over limit!) update_engine tries to write to inactive boot_b: ① writes first 64MB → OK ② tries to write bytes 64–70MB → PARTITION FULL ❌ ③ update_engine returns ERROR: "Failed to write boot partition: ENOSPC" ④ OTA fails entirely. Device stays on current version. ⑤ failure reported to OTA server. # SAFETY: write was to INACTIVE boot_b. boot_a (active) completely untouched. # Device is safe, still running old version. No bricking.
Android's Solutions for boot Size Growth:
Solution 1Split Partitions
If boot is too big → split it into boot + vendor_boot + init_boot. Each physical partition is smaller. Contents distributed across multiple slots. This is Android's PRIMARY strategy since Android 12.
Android 11: boot_a (64MB): kernel+ramdisk+vendor_ramdisk+DTBs Android 12: boot_a (32MB): kernel + generic ramdisk vendor_boot_a (32MB): vendor ramdisk + DTBs init_boot_a (8MB): init ramdisk (Android 13)
Solution 2OEM Over-Provision at Factory
OEM allocates much more space than current boot image needs. "Android grows — give it headroom." Current image: 48MB, partition allocated: 128MB. Future updates have room to grow without hitting the limit.
OEM factory flash: boot_a: 128MB (image currently 48MB — 80MB headroom) boot_b: 128MB (must always equal boot_a size!)
Solution 3Compression
boot.img uses LZ4/GZIP compression for the ramdisk. Kernel itself is compressed (Image.lz4, zImage). Decompressed at boot time by the bootloader. Same content, smaller footprint.
Uncompressed ramdisk: 32MB LZ4 compressed: 12MB ← fits in smaller partition ✓
Solution 4Move Content to /super (Logical)
The architectural trend: move everything possible FROM physical partitions (boot, vendor_boot) TO logical partitions in /super (vendor, odm). Physical partitions = only what MUST be physical (kernel). Everything else → flexible /super.
Goal: boot_a = kernel ONLY (tiny, stable, rarely changes) vendor content → vendor_a (logical in /super, flexible size!) This is the long-term Android architecture direction.
Qualcomm AAOS — DTBO Growth Problem
Why DTBO growsMore peripherals (displays, sensors), new hardware variants, growing CAN/SOME-IP device trees with each AAOS release.
Qualcomm solution 1dtbo partition pre-allocated generously (32–64MB) at factory.
Qualcomm solution 2DTBO overlays moved inside vendor_boot partition.
Qualcomm solution 3Separate dtbo per hardware variant, selected at boot via androidboot.dtbo_idx kernel cmdline parameter.
Architectural Rule: Logical partitions in /super → sizes are FLEXIBLE (grow/shrink/add/remove via LpMetadata). Physical partitions (boot, abl, dtbo) → sizes are FIXED in GPT. The trend is continuously moving content from physical → logical to gain flexibility.
